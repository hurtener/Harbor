# Harbor

A Go-native runtime for durable, steerable, event-driven AI agents.

> Harbor is in pre-V1 development. The design surface is fully landed (RFC, master phase plan, research briefs, contributor normatives); implementation is rolling out wave by wave. See `RFC-001-Harbor.md` for the architecture, `docs/plans/README.md` for the master phase plan (84 V1 + 14 post-V1 phases), and `docs/research/` for phase-planning research briefs.

## At a glance

Harbor is a four-layer system:

- **Harbor Runtime** ÔÇö the orchestration kernel. Tasks, planner runtime, tools, memory, sessions, events, skills.
- **Harbor Protocol** ÔÇö the canonical event/state contract between the Runtime and any client.
- **Harbor Console** ÔÇö the observability + control-plane UI. Architecturally a Protocol client.
- **Harbor CLI** ÔÇö the `harbor` binary. Phase 63 registers the seven settled subcommands (`dev`, `scaffold`, `validate`, `version`, `inspect-events`, `inspect-runs`, `inspect-topology`) with global `--quiet` / `--json` flags. `harbor version` (Phase 63), `harbor scaffold` (Phase 67), `harbor validate` (Phase 68), `harbor dev` (Phase 64), and `harbor inspect-events` / `harbor inspect-runs` (Phase 69) are fully implemented; `harbor validate [path]` runs the in-process config loader against a YAML file (default `harbor.yaml`) without booting any subsystem and emits stable, file:line-precise errors (suitable as a CI pre-flight check); `harbor dev` boots the local Runtime + Protocol server on `127.0.0.1:18080` against `harbor.yaml`, mints an ephemeral ES256 dev-token printed to stderr, mounts the Phase 60 SSE+REST mux behind the Phase 61 JWT validator, and serves until SIGINT/SIGTERM; `harbor inspect-events` tails or snapshots the Phase 60 SSE event stream (identity-scoped, with optional `--type`/`--run`/`--since` filters, human or `--json` NDJSON output) and `harbor inspect-runs` derives per-run summaries (list mode) or per-event trajectories (single-run mode) from the same SSE replay ÔÇö both use a Bearer JWT from `HARBOR_TOKEN` or `~/.harbor/token`. The remaining `inspect-topology` subcommand stub-fails non-zero with a structured `{error,code,hint}` JSON pointing to its implementing phase. Subsequent phases bring hot-reload (Phase 65), draft saving (Phase 66), and the ASCII topology renderer (Phase 70).

Three product properties are non-negotiable: multi-isolation across `(tenant, user, session)`, the Console-as-Protocol-client decoupling, and a swappable Planner.

## Testing your Harbor agent

The public `harbortest/` package gives end-users a five-function authoring surface for flow-level tests ÔÇö `RunOnce`, `AssertSequence`, `AssertNoLeaks`, `SimulateFailure`, `RecordedEvents`. Import as `github.com/hurtener/Harbor/harbortest`; the package's godoc documents the surface, and `harbortest/agent_test.go` is the worked example.

To scaffold a fresh agent project that includes a `harbortest`-driven test, run `bin/harbor scaffold --name my-agent`. The output ships a production-shaped `harbor.yaml`, a `go.mod`, and a worked `EchoAgent` + test pair ÔÇö see Phase 67 above.

The Harbor Console is e2e-tested with Playwright. The harness baseline lives at `web/console/tests/`; `web/console/tests/README.md` is the per-page-spec authoring guide. The suite runs against the `harbor console` subcommand and is gated by the `frontend-e2e` CI job ÔÇö see Phase 75 above.
